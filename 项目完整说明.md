本文档基于当前仓库代码整理，按“1. 后端、2. 前端、3. 脚本与数据流程、4. 项目整体调用关系”四部分进行逐文件精读总结。

## 1. 后端完整说明

### 1.1 `backend/app.py`

这是 FastAPI 的应用入口。

- 创建了 `FastAPI` 实例
- 配置了 CORS，只允许本地前端开发地址访问
- 挂载了三个核心路由：
  - `graph_router`
  - `ai_router`
  - `admin_router`
- 根路径 `/` 返回 API 简介与文档地址

**作用总结**：它是整个后端服务的总入口，负责把图谱查询、AI 服务、管理功能统一暴露给前端。

---

### 1.2 `backend/routers/graph.py`

这是知识图谱查询层的核心接口文件。

从当前项目使用方式看，它主要承担以下职责：

- 健康检查
- 统计信息
- 章节列表与章节详情
- 知识点搜索与详情
- 知识点前驱、后继、相关关系查询
- 图谱整体数据返回

#### 接口语义概览

- `/api/health`：后端存活检查
- `/api/stats`：返回知识点、章节、关系等统计
- `/api/chapters`：章节列表
- `/api/chapters/{chapter_id}`：章节详情
- `/api/kps/search?q=...`：搜索知识点
- `/api/kps/{kp_id}`：知识点详情
- `/api/kps/{kp_id}/prerequisites`：先修知识点
- `/api/kps/{kp_id}/successors`：后继知识点
- `/api/kps/{kp_id}/related`：相关知识点
- `/api/graph`：图谱可视化所需节点和边数据

#### 核心特点

- 直接基于 Neo4j 查询
- 返回结构化数据，方便前端图谱和详情面板渲染
- 关系查询符合学习路径语义：先修、后继、相关

**作用总结**：它是前端所有“看图谱”的数据来源。

---

### 1.3 `backend/routers/ai.py`

这是 AI 能力的路由封装层。

#### 提供的能力

- AI 服务健康检查
- 聊天问答
- 流式聊天输出
- 知识点解释
- 学习路径推荐
- 代码审查

#### 主要接口

- `GET /api/ai/health`
- `POST /api/ai/chat`
- `GET /api/ai/chat/stream`
- `POST /api/ai/explain`
- `POST /api/ai/recommend-path`
- `POST /api/ai/code-review`

#### 设计特点

- 路由层不直接拼提示词，而是调用 `ai_service`
- 支持普通返回和流式返回
- 对知识点解释和推荐路径进行了针对性封装

**作用总结**：它把 AI 功能从通用聊天扩展成了“教学助手”能力。

---

### 1.4 `backend/routers/admin.py`

这是整个系统最复杂的模块之一，负责管理端数据流转。

它的核心功能可以分成四类：

#### A. PDF 上传与抽取

- 上传 PDF
- 创建任务目录
- 启动异步抽取线程
- 读取 PDF 文本，必要时 OCR 回退
- 使用文本清洗管道
- 调用 AI 抽取章节、知识点和关系
- 结果修复、规范化、容错
- 支持重试和 fallback

#### B. 文件导入预览

- CSV/JSON 文件解析
- 识别章节、知识点、关系
- 返回预览数据供前端确认

#### C. 审核队列

- 把导入数据放入内存审核队列
- 支持 approve / reject
- 支持批量通过
- 支持统一应用到 Neo4j

#### D. 知识点手动管理

- 手动新增知识点
- 更新知识点
- 删除知识点
- 这些操作先进入审核队列，再落库

#### 关键实现细节

- 使用 `fitz` 读取 PDF
- 使用 `pytesseract` 进行 OCR 回退
- 使用 `clean_pdf_text` 清洗文本
- 使用 AI 返回 JSON 后做多层容错修复
- 使用 `_coerce_payload_to_graph_schema` 统一不同 AI 输出结构
- 使用 `_fallback_extract_from_text` 在 AI 失败时兜底
- 审核数据先存内存队列 `_review_queue`

#### 任务存储结构

管理端会在 `data/uploads/` 下维护：

- PDF 上传文件
- job 任务目录
- 状态文件 `status.json`
- 抽取中间结果
- 扫描日志
- 清洗结果
- prompt 与 AI 原始输出

**作用总结**：这是项目从“上传资料”到“形成图谱数据”的主入口。

---

### 1.5 `backend/db/neo4j.py`

这是 Neo4j 数据访问层。

#### 主要职责

- 从 `.env` 读取 Neo4j 连接参数
- 创建 Neo4j driver
- 提供统一的 `run(query, params)` 方法
- 返回字典列表结果

#### 特点

- 启动时即检查环境变量
- 使用 `GraphDatabase.driver`
- 所有查询统一走一个薄封装

**作用总结**：它是后端所有图数据库操作的底座。

---

### 1.6 `backend/models/schemas.py`

这是后端数据返回结构定义文件。

当前主要定义：

- `ChapterOut`
- `KnowledgePointOut`
- `ChapterDetailOut`

#### 表达的信息

- 章节包含 `chapter_id / title / order`
- 知识点包含来源、章节、小节、别名等字段
- 章节详情会把章节和其下知识点一起返回

**作用总结**：它让后端输出数据更稳定、可预测。

---

### 1.7 `backend/services/ai.py`

这是 AI 服务核心封装层。

#### 支持的 provider

- `ollama`
- `openai` 兼容接口

#### 核心内容

- 系统提示词 `SYSTEM_PROMPT`
- `OllamaClient`
- `OpenAICompatibleClient`
- `AIService`

#### 主要能力

- 判断 AI 服务是否可用
- 生成文本
- 流式输出
- 知识点解释
- 学习路径推荐
- 代码审查

#### 设计亮点

- 对 OpenAI 兼容接口做了 fallback model 支持
- 对流式响应进行了专门解析
- `AIService` 统一对外暴露接口，路由层无需关心供应商差异

**作用总结**：它把不同 AI 后端统一成一个教学助手接口。

## 3. 脚本与数据流程完整说明

### 3.1 `backend/scripts/course_schema.py`

这是课程知识图谱的“种子数据定义文件”。

#### 内容包含

- 章节列表 `CHAPTERS`
- 知识点列表 `KNOWLEDGE_POINTS`

## 4. 项目整体调用关系说明

下面按真实业务流程串起来。

### 4.1 初始化阶段

1. 启动 Neo4j
2. 配置 `.env`
3. 启动后端 `FastAPI`
4. 前端 `Vite` 启动
5. 前端加载章节列表和健康检查

### 4.2 图谱浏览流程

1. 前端请求 `/api/chapters`
2. 请求 `/api/stats`
3. 进入图谱页后请求 `/api/graph`
4. 点击知识点后请求：
   - `/api/kps/{kp_id}`
   - `/api/kps/{kp_id}/prerequisites`
   - `/api/kps/{kp_id}/successors`
   - `/api/kps/{kp_id}/related`
5. 右侧详情面板展示知识点信息

### 4.3 AI 学习流程

1. 用户打开聊天抽屉
2. 前端调用 `/api/ai/chat` 或流式接口
3. 后端调用 `ai_service`
4. 返回回答、解释、路径建议或代码审查结果

### 4.4 管理导入流程

1. 上传 PDF 或数据文件
2. 后端创建任务目录
3. 读取 PDF / 解析文件
4. 清洗文本
5. 调用 AI 抽取结构化结果
6. 做 fallback 和修复
7. 把结果放入审核队列
8. 管理员批准后应用到 Neo4j

### 4.5 数据骨架导入流程

1. 运行 `seed_course.py`
2. 先建立 Neo4j 约束
3. 再导入章节、知识点、示例、关系
4. 图谱系统立刻具备完整基础数据
